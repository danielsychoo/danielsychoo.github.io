Cucumber를 통한 테스트 자동화

Cucumber는 사용자친화적으로 테스트자동화를 구축할 수 있는 프레임워크(?)라고 한다. 이를 회사에서 구축 및 적용해보자고 하여 공식문서를 보며 공부하는 과정을 서술하려 한다.

Cucumber 설치 및 환경설정

Cucumber를 사용할 디렉토리 내에서 npm initialize를 진행하고 Cucumber를 설치한다.

이후 package.json에서 scripts에 cucumber를 실행하도록 작성한다. 이로써 기본적인 사용 준비는 끝.

터미널에서 npm test를 실행했을 때 위와 같이 나오면 준비가 완료된 것이다.

다음으로 더 다양한 assertions 를 사용하기 위해
hamjest를 설치한다. assert란 우리가 작성한 로직과 함수를 검증하기 위한 검증 구문이라고 생각하면 되는데, 이러한 구문들의 집합체를 assertions 라고 부른다.

assertion 라이브러리는 여러 가지가 있으나, 우선은 cucumber-school에서 권장하는 hamjest로 설치했다.

시나리오 추가 및 연결하기

실제적으로 시나리오를 작성하고 추가해보기 위해 features 디렉토리를 만들고 해당 디렉토리의 내부에 hear_shout.feature 파일을 생성한다.

이제 Gherkin에 대한 것들은 모두 features 디렉토리에 저장할 것이다. Gherkin이란 Cucumber에서 test를 작성하는 문법이다.

문법 작성 후 npm test를 실행하면 해당 테스트의 실행결과와 함께 녹색 박스로 cucumber의 report를 공유할 수 있다고 하는데, 현재는 해당 기능을 사용할 의사가 없으므로,

cucumber.js 파일에 quiet 옵션을 적용한다.

시나리오 단계 실행을 위한 단계정의

해당 녹색 박스를 제외한 test summery를 확인하면 세 단계가 모두 undefined임을 확인할 수 있음. 이유는 해당 단계들 중 어떤 것도 수행할 작업을 찾지 못했기 때문. 따라서 단계에 대한 정의를 해야하는데, 이 단계 정의는 Gherkin에서 사용하는 일반 언어를 JavaScript로 변환되도록 하는 것임.

즉, Cucumber는 feature에서 단계를 찾으면 해당 단계에 대해 정의된 코드를 실행한다는 것. (예로 GraphQl에서 feature가 query이면, 단계 정의는 resolver로 볼 수 있음) 게다가 터미널의 오류 내역을 보면 어떤 식으로 단계정의를 해야할지 스니펫도 안내해줌.

따라서 feature파일에 작성한 위 테스트 문장을 steps.js 파일에 정의해야 한다.

이후 다시 npm test를 실행하면 첫 번째 단계는 pending, 나머지 두 단계는 skip 됨을 볼 수 있다.
여기서 Given 부분만 수정하여 int의 값을 알아보자.

int에 해당하는 값을 조금 더 알아보기 좋게 distance라고 지칭하고, 해당 내용을 console.log로 찍어보면 15라는 값이 나온다. 즉 Given의 첫 인자 String 안에있는 {int}는 feature파일에서 해당 위치에 존재하는 값의 자료형을 의미하는 것이고, 이 값을 두 번째 인자에서 parameter로 명명하여 사용한다.

여기까지를 통해 바라본 단계 정의의 정리

1) 단계 정의의 기능은 무엇일까 ?

우리가 Gherkin 시나리오에 작성한 단계들을 Cucumber가 자동화 할 수 있도록 하는 것.

자바스크립트에서 단계 정의는 Gherkin 시나리오의 각 단계를 실제로 수행하는 함수이다. Cucumber는 시나리오의 각 단계를 실행하려고 할 때 그 단계와 일치하는 단계정의를 검색하고, 일치하는 정의가 있다면 해당 함수를 호출한다.

2) Cucumber에서 단계가 pending 상태인 것은 무슨 의미일까 ?

Cucumber가 pending을 반환한다는 것은 해당 단계의 테스트 자동화가 pending 상태라는 것이다. 이를 통해 우리는 현재 해당 단계의 테스트 자동화 작업이 이루어지고 있음을 나타낼 수 있다. (만일 자동화 작업 계획이 없는 부분이라면 pending 조차 반환하지 않으므로)

3) Cucumber에서 생성된 스니펫을 통해 고려할 수 있는 부분

단계정의를 하지 않은 상황에서 npm test를 할 경우 나타나는 스니펫을 통해 우리는 해당 테스트의 이름이 우리의 목적과 부합하는지, parameter의 이름이 우리가 의도하는 의미와 부합되는지 고려해볼 수 있다.

4) 나타난 스니펫 사용 후 pending 실패를 해결하기 위한 다음 단계는 ?

Gherkin 시나리오에서 설명하는 작업을 수행하는 코드를 구현할 차례이다. 스니펫에서 생성된 주석을 통해 무엇을 조치해야 하는지 구체적으로 알 수 있다. 바로 이러한 방식으로 인해 솔루션을 구현할 때 실제 요구 사항을 기반으로 구현하는 데 도움이 된다.

분석

이로인한 문제로, 어떤 기능을 개발할 때 해당 기능이 다른 버전에 존재하는지를 파악하는데도 오래걸릴 뿐더러 추후 이 상황이 오래되면 통합하는데도 오랜 시간이 걸릴 것이라 예상 되었고,

결국 팀 내에서 PC, AOS, IOS 각 파트별로 버전별 서비스의 기능을 정리하고 어떻게 통합할지에 대해 회의하기로 했다.

사실 나의 경우에는 덕분에 전체적으로 회사의 서비스가 어떤 구조로 되어있고, 그 코드는 어떻게 되어있는지 파악할 좋은 기회라고 생각했다.

하지만 생각보다 버전별로 기능이 많이 달라서 이를 따라가는 것도 벅찼다. 이에 각 버전을 만들게 된 계기를 기준으로 어떤 기능적 차이가 있는지 확인했다.

결론

결과적으로 분석한 자료를 가지고 팀내에서 회의를 해봐야겠지만, 이번 분석을 통해 느낀점은 만들기 전에 충분히 생각해봐야 한다는 점이었다.

어떤 서비스의 분기를 다루거나 기능을 추가할 때 해당 버전을 하나로 통합할 수 있는지, 이로인해 중복되는 서비스가 생기지는 않는지를 미리 생각해보고 개발을 진행해야 이러한 문제가 없을 것으로 보인다.

apollo의 local state

apollo에서는 local state를 지원하는데, 이 local state는 전역관리가 가능하다.

이를 이용하기 위해서는 initialState를 만들고, 이 state를 업데이트하기 위한 resolvers를 작성한 뒤 state를 사용할 부분에서 query 또는 mutation에 @client를 넣는 방식으로 이루어진다.

1) initialState

해당 예시는 board의 index를 관리하기위한 state

2) resolvers

3) 실제로 local state를 사용하는 mutation

이 경우 server와의 통신을 통한 data를 함께 query 또는 mutation으로 사용할 수 있으며, local state를 사용할 부분에만 @client를 넣어주는 방식으로 사용한다.

apollo의 fetch policy

apollo는 fetchPolicy를 설정하여 데이터 캐시와의 상호작용 방식을 지정할 수 있다. apollo의 default fetchPolicy는 catch-first방식으로 flow는 다음과 같다.

1. 쿼리의 데이터가 캐시에 있으면 캐시에서 데이터 반환 (쿼리 전체 중 있는 것만이라도)
2. 캐시에 없는 부분이 있다면, 네트워크를 사용하여 데이터 요청을 실행

fetchPolicy를 custom한다면 이와같은 상호작용 방식을 변경할 수 있다. custom할 수 있는 종류는 아래와 같다.

1) cache-first

항상 캐시를 먼저 확인한다. 확인 후 캐시에 없는 데이터는 network 요청을 보낸 후 res를 통해 update된 data를 반환한다.

2) network-only

캐시를 확인하지 않고 항상 network 요청 후 res를 반환한다. (이때 캐시에 업데이트는 이루어진다.)

3) cache-only

캐시만 확인한다. 만일 캐시에 데이터가 없다면 error를 반환한다.

4) cache-and-network

먼저 캐시를 확인하여 데이터를 반환한다. 이때, 캐시에 있는 데이터가 업데이트 되지 않은 경우라도 우선적인 반환이 이루어진다.

이후 network 요청 하여 새로운 data를 받아 캐시를 업데이트하고, 업데이트 된 data를 한번 더 반환한다.

5) no-cache

캐시를 확인하지 않고 항상 network 요청을 보낸다. 2번 network-only와의 차이점은 쿼리가 완료된 후 캐시의 data를 읽지 않는다는 점이다.

Container Presenter DesignPattern (2021-05-07 금)

container-presenter 디자인 패턴

container-presenter 디자인 패턴이란 한 기능의 디렉토리 내부에 기능을 담당하는 js파일과 디자인을 담당하는 scss파일이 존재하는 디자인 패턴을 말한다.

우리의 경우 각 기능 디렉토리별로 index.js까지 추가하여 해당 component가 index.js를 바라보도록 한다.

utils

사용되는 함수나 상수 등이 길어질 경우 Component 내에서 선언하고 사용되면 지저분해질 수 있으므로 utils 디렉토리내에 선언하고 꺼내쓰는 방식을 이용한다.

  • src
    • utils
      • router.js: route 모음
      • words.js: user에게 보여줄 문자열 모음
      • authUtil.js: auth에 사용될 함수모음
      • …

constant.js

고정적으로 사용되는 상수나 문자열의 경우에 일일히 하드코딩을 하기보다 constant.js에 몰아넣고 꺼내쓰는 방식으로 관리한다.

이렇게 관리할 경우 상수나 문자열 수정시 일일히 수정할 필요 없이 constant.js만 수정해줌으로서 수정된 결과물을 얻을 수 있다는 장점이 있다.

이런식으로 나누어진 전체 패턴을 살펴보면 다음과 같다.

    • Apollo

webpack과 globalStyles.scss

프로젝트 파일을 살피던 중 scss의 이 장점을 조금 더 극대화하여 사용할 수 있는 방법을 발견했다. 이는 webpack과 globalStyles.scss를 이용하는 방법이다. 절차적으로보면 다음과 같다.

  1. globalStyles.scss에 사용할 변수와 mixin된 부분 등을 import한다.
  2. webpack에서 config를 통해 globalStyles.scss가 전체 적용되도록 한다.

이 과정을 통해 얻을 수 있는 이점은 전체 scss파일에 globalStyles.scss가 적용되도록 해두었으므로, 이후 작업하는 각각의 scss파일에
@import를 진행할 필요가 없다는 것이다.

react loading(2021-05-04 화)

loading 기능

일반적으로 user의 event로 인해 통신이 일어날 경우 그 결과가 나타나기 전까지 loading page를 출력한다.

사실 이 페이지가 별 것 아닌 것 같지만 user의 입장에서 loading을 보고 기다리는 것과 새하얀 백지를 보며 멈춘 것인지 아닌지 확인을 못하는 것의 차이는 크다고 볼 수 있다.

apollo-graphql에서의 Loading

사실 apollo-graphql의 스택을 이용할 경우 쿼리를 이용했을때 loading 자체를 훅으로 뽑아쓸 수 있기 때문에 따로 무언가를 처리한다거나 할 필요는 없다.

RestAPI에서의 Loading

그러나 이러한 훅이 없는 RestAPI 등에서는 비동기를 이용해 Loading을 처리해야 한다. 우선 비동기를 통한 flow는 아래와 같이 볼 수 있다.

  1. { isLoading: false } state 선언
  2. axios를 실행하기 전에 isLoading: true
  3. axios통신 후 isLoading: false

이때 isLoading이 true이면 로딩을 보여주고 false이면 다시 돌아오는 방식으로 JSX는 작성된다.

react에서 loading관련 라이브러리

react에서 loading을 출력하는 라이브러리를 선택할 때에는 우선 어떤 모양새로 loading을 출력할지 확인해야 한다.

구글링 결과 spinner방식을 사용할 때에는 react-loader-spinner 라이브러리를, progressbar 방식을 사용할 때에는 react-loading 라이브러리를 많이 사용한다는 것을 발견했다.

라이브러리는 모양을 출력해주는 것이 전부이기 때문에 해당 라이브러리를 이용해 Loading.js 컴포넌트를 구성하고, 위의 로직에 따라 해당 컴포넌트를 조작하는 방식으로 구현한다.

proxy를 통한 CORS 해결 (2021-04-21 수)

CORS 에러

4월 19일 월요일부터 게시판을 만드는 mini-project를 시작했는데, CORS에러와 함께 client side에서 cookie를 받아오지 못하는 등의 에러들로 고생을 했다.

서버는 RestAPI로 구축이 되어있는 상황에서 API 명세서를 확인하며 axios로 기능을 구현했으나, CORS에러가 나타나며 로그인 후 cookie를 client side에서 받아오지 못했다.

처음에는 코드의 로직 자체에 문제점이 있다고 판단했으나, 개발자도구를 살펴보니 network탭의 Headers에는 cookie가 잘 실려 오지만, application탭에서는 cookie가 존재하지 않았고, 오늘까지 장장 3일에 걸친 이를 해결하기위한 과정이 시작되었다.

기본적으로 CORS는 위와같이 Server와 Client의 주소가 다를때 나타나는데 이를 해결하기위한 구글의 방법을 찾아본 결과 아래의 방법이 대표적으로 나왔다.

withCredentials: true

이 방법은 cross-site Access-Control 요청에 인증서를 포함시킴으로서 CORS를 해결하는 방법이다. 따라서 client side에서는 위와 같이 true로 해줌으로서 조치할 수 있다.

다만, 이 방법을 사용하려면 server side에서도 credential을 true로 설정해줘야 한다.

proxy 설정

헌데, 위 server side를 고칠 수 없는 상황이라 위 방법을 쓸 수 없다면 어떻게 해야할까? 이럴때 사용할 수 있는 방법이 proxy를 이용하는 것이다.

proxy는 보안상의 이유로 직접 통신할 수 없는 두 점 사이에서 대리로 통신하는 중계역할을 하는 것을 말한다. 따라서 client 입장에서 proxy는 원격 server로 볼 수 있고, server 입장에서는 원격 client로 볼 수 있다.

proxy없이 요청을 하게 될 경우 server와 client의 uri가 다르기에 CORS에러가 발생되지만, proxy가 server와 client사이의 중계기능(가상서버) 역할을 할때

client는 proxy에게 client uri로 그대로 요청을 하고 proxy내부에서 이를 server uri로 변환하여 server로 보내는 방식으로 이루어지기 때문에

요청도 제대로 들어가고 Host와 Origin이 같으므로 CORS문제도 없게된다. (아래사진참조)

cookie 관리하는 방법 (2021-04-20 화)

session과 cookie

로그인한 유저의 인증방식 중 session방식은 로그인한 시간을 기준으로 만료날짜까지의 시간동안 인증이 되는 방식이다. 따라서 로그인을 요청하면 server에서 session을 cookie에 담아준다.

session을 cookie에 담아주면 브라우저에 저장이 되기 때문에, 로그아웃시에 client side에서 이를 delete하는 로직이 필요하다. (나는 client side의 정보를 파기하는 것은 생각치 못했는데, 이로인해 지속적으로 문제가 발생되었었다.)

이를 관리하기 편한 방법은 library를 사용하는 것인데, 구글링을 해보면 대부분 react-cookie 또는 js-cookie를 추천하는 편이다. 나의 경우 js-cookie를 이용했다.

cookie 생성

기본적으로 server에 로그인을 요청하면 cookie를 함께 보내주지만, 혹시나 client side에서 cookie를 생성해야 할 경우 아래의 코드를 이용한다.

cookie를 조회할 경우 아래의 코드를 이용해 간단하게 조회할 수 있다. 이때, 해당 메소드를 구지 props로 넘겨줄 필요 없이(react 기준) 필요 컴포넌트에서 다시 import 해오면 된다.

cookie 삭제

cookie를 삭제할 경우 해당 name을 메소드에 넣어주는 것만으로 삭제가 된다.

cookie에 대해

처음에 나는 cookie로 들어오는 값을 따로 변수에 담아 필요시 이용하려했는데, cookie는 server에 요청을 날릴때 필요하면 자동으로 날아간다.

react에서 파일올리기 기능 구현 (2021-04-16 금)

file upload 기능

서비스 및 어플리케이션을 만들다보면 파일을 업로드하는 기능이 필요할 때가 있다. 헌데, 파일을 업로드하는 과정은 어떻게 이루어지는지를 전혀 몰라서 찾아보게 되었다.

결과적으로 보았을 때, 우리가 input의 내용을 local state에 담아 axios.post를 통해 서버로 보내는 방식은 동일한데, 그저 보내는 data의 타입이 text에서 file로 바뀐 것일 뿐으로 보인다.

input type=”file”

평소 회원가입, 로그인 등의 기능을 구현하는 과정에서 input의 type="text"로 했던 부분을 type="file로 하면 file을 담을 수 있는 input이 된다.

1) accept

input의 메소드로 accept를 사용하면 user가 입력할 수 있는 파일의 유형을 선택할 수 있다. 그러나 이 것은 강제는 아니고, 입력할때의 default option을 정해주는 것이다. 따라서 파일을 입력할때 user가 모든 파일 (*.*)로 변경할 수 있다.

2) multiple

input의 메소드로 multiple을 사용하면 파일 선택시 여러개의 파일을 선택할 수 있다. default는 false로 하나의 파일만 선택이 가능하다. 이 경우에는 state에 file 하나가 아닌 file들이 담겨있는 배열이 들어가므로 이를 감안한 코드 작성이 필요하다.

new FormData()

일반적으로 form 태그를 사용하게 될 경우 onClick시에 페이지 전환이 일어나므로 event.preventDefault()를 이용해 전환을 막아야 한다.

이런 번거로운 부분을 피하고, 페이지 전환이 없이 form data를 보내고 싶을때 사용하는 것이 FormData이다.

1) formData에 data 넣기

formData(instance)는 append 메소드를 사용해 key-value쌍을 추가할 수 있다. 이때 key는 중복이 되어도 value가 바뀌지 않으며, 동일한 key에 새로운 value로 추가되는 방식으로 동작한다.

2) formData의 data 확인하기

formData(instance)에 들어있는 data는 특정 key가 존재하는지 확인, key에 해당하는 첫번째 value 확인, key에 해당하는 모든 value를 확인. 의 세 가지 방법을 사용해 확인할 수 있다.

React의 memoizing 기능들 (2021-04-14 수)

성능향상을 위한 memoizing 기능들

React.memo()

React는 component를 rendering한 뒤 이전 rendering과 비교하여 DOM의 업데이트를 결정한다. 만일 component를 React.memo()로 래핑한다면, React는 해당 component를 rendering하고, 그 결과를 memoizing한다. 그리고 props의 변화가 일어나기 전까지 memoizing된 내용을 재사용한다.

이때 주의할 점은 `props에 대한 부분만 해당될 뿐 state에 대한 변화는

이는 class형의 componentLifeCycle중 shouldComponentUpdate에 대응되는 기능으로, return되는 값이 true이면 re-rendering이 이루어 진다.

React.memo의 비교방식 custom

기본적으로 React.memo()는 shallow copy를 통해 props를 비교하는 것이 기본동작이다. 만일 custom한 비교를 원한다면, 두 번째 인자에 custom한 비교 함수를 제공하는 방식을 사용한다.

위 코드와 같이 여러방식의 custom한 비교방식을 통해 비교를 할 수 있다. 이때 기억해야할 점은 true를 반환하면 re-rendering이라는 점이다.

React.memo의 사용이 효율적인 상황

React.memo를 사용하는 것이 반드시 성능을 효율적으로 만들어주지는 않는다.

  • rendering시에 props가 다른경우
  • class형 component를 사용하는 경우 (shouldComponentUpdate를 사용)

예로 위 두 상황의 경우에는 props가 달라 memoizing의 의미가 없는 등의 상황으로 쓸모없는 props 비교가 이루어진다.

따라서 React.memo는 같은 props를 이용해 자주 rendering될 것으로 예상되는 부분에 사용하는 것이 가장 좋은 성능을 발휘한다. (예로 게시판에 게시글 1000개 중 특정 1개를 수정했을 경우, 나머지 999개는 re-rendering될 필요가 없게끔)

이를 정 판단하기 어렵다면 react dev-tool의 profiler를 이용해 비교해보는 것이 나을 듯…

useCallback unnecessary dependency

useCallback 에서 나타나는 오류 중 unnecessary dependency 경고가 콘솔에 뜨면, second parameter에 쓸모없는 값이 들어가있다는 것이다.

Line 9:5: React Hook useCallback has an unnecessary dependency: 'state'.

나의 경우 계속해서 이런 경고가 발생되었는데, 알고보니 함수 안에서 사용하는 state 또는 props를 내가 잘못 이해했기 때문이었다. 이 어휘는 말 그대로 useCallback 안에서 사용하는 것을 의미한다.

GraphQL yoga 공부 (2021-04-09 금)

GraphQL-yoga

GraphQL로 해결할 수 있는 문제들

  1. Over-fetching: 요청에 대해 필요한 data 이상으로 data를 받아와 비효율적인 상황
  2. Under-fetching: 필요한 여러 data를 위해 여러번 요청해야 하는 비효율적인 상황

GraphQL은 두 가지의 특징에서 이러한 부분들을 해결할 수 있음.

  1. 요청을 하는 endpoint가 하나로 되어있음. 따라서 각각의 data를 위해 각각 요청할 필요가 없음.
  2. front에서 직접 query를 작성하여 필요한 정보만 받아올 수 있음.

React router 공부 (2021-04-08 목)

공부한 내용: React-router

두 가지 종류

1) HashRouter

주소에 해쉬(#)가 붙고, 검색엔진이 읽지 못함. 정적 페이지에 적합함.

2) BrowserRouter

새로고침을 하면 경로를 못찾아 에러가 발생됨. 따라서 a태그가 아닌 Link태그를 사용. 동적 페이지에 적합함.

따라서 BrowserRouter의 사용을 권장함.

exact path

route를 설정할 때 경로를 설정하는 방법은 path와 exact path의 방법이 있음. 이때 exact를 붙여야 정확히 맞아떨어지는 경로로 이동됨. 아래의 예시 참조

Switch

중복되는 경로를 구분하는 방법은 exact path외에도 Switch의 방법이 있음. 이 경우 Switch내에 있는 컴포넌트들 중 매칭되는 첫 번째 컴포넌트만 보여주게 됨.

따라서 먼저 비교할 라우트를 위에 작성해야 함.

history, location, match

1) history

history의 push, replace를 통해 다른 경로로 이동하거나 앞 뒤 페이지로 전환 가능

2) location

location은 현재 경로에 대한 정보를 지니고 있고, URL 쿼리 (예로 /about?foo=bar) 정보를 가지고 있음

3) match

match는 어떤 라우트에 매칭이 되었는지에 대한 정보를 지니고 있고, params (예로 /about/:name) 정보를 가지고 있음

라우트 경로에 특정 값을 넣어 분기할 경우

1) params (match)

route를 설정할때 path에 /:name과 같이 넣어주면 match.params로 해당 값(이 예시에서는 name)이 들어감. 이를 통한 경로 분기

2) URL 쿼리 (location)

우선 쿼리로 전달된 데이터를 string에서 json으로 파싱하기위해 query-string라이브러리 설치 (이외에도 여러가지 파싱방법이 있음).

location.search의 정보를 위에서 설치한 library인 queryString을 통해 파싱한 후 해당 내용의 .datail을 비교하여 분기. 아래 예시코드 참조

custom hooks 공부 (2021-04-07 수)

공부한 내용: useReducer, custom Hooks, scss

useReducer

1) reducer 함수

state와 action을 parameter로 받아와서 action에 해당하는 조치 후 nextState를 반환하는 함수

2) action

action이란 업데이트를 위한 정보를 의미

3) useReducer

일반적으로 useReducer를 사용하기위해 개발하는 순서

  1. 해당 component에서 사용할 initialState를 component외부 스코프에 선언
  2. reducer 함수의 틀만 만들어주고 component에 useReducer 사용
  3. state에서 필요한 값들을 destructuring해서 각 component에 전달
  4. action 및 reducer 함수 코드 작성

Custom Hooks

  • 반복되는 로직을 custom hooks으로 분리해 재사용 할 수 있도록 하려는 목적
  • 일반적으로 src/hooks에 모아놓음

따라서 custom hooks는

  1. Custom Hooks 안에서 useState, useEffect, useReducer, useCallback 등을 사용하여 원하는 기능을 구현한 후
  2. component에서 구현한 custom hooks를 사용해 원하는 값을 반환

하는 구조로 이루어진다. custom hooks를 만들고나면 결국 [currentValue, setCurrentValue] = customHook();의 형태가 되고, setCurrentValue가 custom되는 것이다.

SCSS

  • scss는 css의 pre-processor이다.
  • 따라서 scss는 typescript를 javascript로 compile하는 것과 같이 css로 compile하는 과정이 필요하다.
  • 그러나 react 에서는 compile의 과정이 필요하지 않다.
  • scss의 주요 기능들
    • 중첩(Nesting)
    • 상위 선택자 참조(Ampersand): &
    • 중첩 벗어나기: @at-root
    • 변수(variables): $
      • 이때, 변수의 유효범위는 Block scope에 따른다.
    • 믹스인(Mixin): Mixin 믹스인이름($parameter) {스타일;}
      • 설정 후 사용시: @include 믹스인이름(parameter)
    • 조건문(if)

결과적으로 css보다 다양한 기능을 js처럼 쓰고, compile을 통해 css로 변환하는 css의 superset.

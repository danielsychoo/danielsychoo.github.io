미니프로젝트(게시판) 결과 (2021-04-30 금)

미니프로젝트 (게시판)

회사에 입사 후 2주간의 공부, 이후 2주간의 미니프로젝트를 통한 테스트를 진행했는데, 프로젝트는 react와 Restful API로 게시판을 만드는 것이었다.

게시판은 CRUD를 공부할 좋은 기회이기도 하고, 입사 전 공부한 포트폴리오 자체가 큰 게시판(웹소설 서비스)의 형태였기에 별 힘듦이 없을 줄 알았으나, 생각보다 복잡했고, 이에 메모하며 진행한 로직들을 정리해보려 한다.

정리한 전체 내용은 위 사진과 같으며, 좌상단부터 시계방향으로 아래와 같다.

  1. (좌상) 게시글 CRUD로직
  2. (우상) 프로젝트 Design Pattern
  3. (우하) 댓글 수정 로직
  4. (중하) 댓글 CRUD 로직
  5. (좌하) 게시글 수정 로직

게시글, 댓글 로직

본 로직을 마인드맵 형식으로 정리하게 된 계기는 user가 게시판을 이용하며 마주할 수 있는 여러 상황들을 빠뜨리지 않기 위함이다.

예로 게시글의 글만 등록하는 상황과 파일을 함께 등록하는 상황은 분리하여 생각해야 하는데, 이런 부분을 미리 체크하지 못해 다시 코드를 뒤엎는 참사를 막기위해 먼저 정리를 하는 것이 중요했다.

또한, 잠시 쉬고오거나 퇴근 후 다음날이 되었을 때 어디까지 진행했고 어디부터 시작해야 하는지를 파악하느라 소모되는 시간을 줄일 수도 있었다. (완료되지 않거나 검색해야 하는 부분을 형광펜으로 표시)

생각보다 이렇게 정리하고 진행하는 것이 큰 도움이 되어 후에 댓글에 대한 부분을 진행할때도 같은 방식으로 정리 후 진행했다.

Design Pattern

이전에 포트폴리오를 준비하면서 했던 프로젝트는 하나의 기능을 담당하는 component와 scss를 하나의 디렉토리로 묶어 관리했는데, 후에 component가 어디로 이어지는지와 data의 전달을 파악하는 과정에서 애를 먹었다.

특히 리펙토링 과정에서 스스로가 짠 로직임에도 데이터를 놓치는 경우가 많았다. 이에 조금 더 나은 방향으로 design pattern을 바꿀 계획을 가지게 되었고 이번 프로젝트에 적용해보았다.

우선 App.js에서 전체 route를 분기해주고, route 디렉토리에 route로 분기된 component들을 넣어준다. 이 component들이 각 page를 구성하게 되고, 내부에 view를 담당하는 component들은 component 디렉토리에 모아준다.

다음으로 각 기능들을 담당하는 custom hook들은 hooks 디렉토리에 모아놓고 필요한 기능들만 꺼내쓰는 방식으로 진행한다. 마지막으로 디자인을 담당하는 scss 디렉토리를 만들어 디자인 파일만을 모아놓는다.

  • index.js
  • app.js
  • route dir
    • route되는 component들
  • component dir
    • view 담당 component들
  • hooks dir
    • 기능을 모아놓은 hook들
  • scss dir
    • 디자인 관련 파일들

복잡한 수정기능

사실 모든 기능들 중에 가장 복잡한 기능은 수정기능이었다. 아무래도 생성은 input을 받아 axios.post로 보내주면 되고, 출력은 axios.get으로 받아오면 되는데,

수정은 수정할 data를 넘겨받아 해당 data를 input에 출력 후 user가 수정하는 data를 onChange로 받은 후 다시금 axios.post로 보내줘야 하기 때문에 신경쓸 부분이 많았다.

게시글의 경우 첨부파일이 존재하는 것과 존재하지 않는 것에 따라 formData의 형식과 header가 달라졌기에 이 부분을 중점적으로 신경쓰게 되었고,

댓글의 경우 수정하는 부분과 출력하는 부분이 동등한 위계의 component에서 진행되었기에 부모 component를 통한 필요 data의 이동에 집중하게 되었다.

추후에도 프로젝트 진행시 이와같이 그려보는 것을 이어가봐야겠다.

react devtool을 통한 성능확인 (2021-04-29 목)

react 성능

react로 만들어진 웹의 성능은 리렌더링시에 불필요한 정보 및 컴포넌트의 리렌더를 얼마나 줄이느냐가 관건이다. 이 부분은 Chrome extention중 React DevTools의 profiler를 사용하여 확인 및 테스트할 수 있다.

Profiler

React Devtools중 Components 탭은 컴포넌트 사이에서 props가 잘 전달되고 있는지, 혹은 액션에 따라 state가 잘 관리되고 있는지 등을 확인하기 위해 앱을 구현하는 과정에서 자주 확인하게 되는데, profiler 탭의 경우에는 구현이 된 이후 성능 및 속도등의 부분을 확인하기 위해 사용된다.

사실상 profiler는 앱의 활동과정을 녹화한 후 그 녹화 중에 내부에서 벌어지는 렌더링을 잡아내고 시간이 얼마나 걸리는지 확인해주는 방식으로 동작한다. 따라서 녹화버튼을 누른 후 테스트할 페이지를 수동으로 들어가야 한다.

  1. profiler 탭의 녹화버튼 누르고
  2. 테스트할 페이지 렌더링 실시
  3. 다시 녹화버튼 눌러 녹화를 끄고
  4. 결과 확인 및 분석

profiler 결과창 분석

profiler의 결과창에서 우리가 파악할 수 있는 부분은 아래 그림과 같다.

먼저, 회색 으로 나타나는 컴포넌트는 녹화과정에서 렌더링 된 적이 없는 부분이다. 따라서 리렌더링을 원치 않는 부분이 회색으로 나타나지 않는다면 조치가 필요하다.

Router의 경우 Router가 설정되어있는 컴포넌트가 첫 렌더될 시에만 렌더링 되고, 이후에는 늘 리렌더링이 되지 않는다.

반대로 색상이 들어간 부분은 렌더링이 일어난 부분이다. 해당 그림중 Navigation 컴포넌트에 보면 0.3ms of 1.5ms라는 부분이 있는데, 1.5ms라는 이 페이지의 전체 렌더링 시간 중 Navigation 컴포넌트의 렌더링 시간이 0.3ms만큼 잡아먹었다는 뜻이다.

이를 통해 내부의 컴포넌트들 중 어느 컴포넌트의 성능이 떨어지는지를 확인할 수도 있다. 확인 후 해당 컴포넌트에서 다시한번 녹화를 한다면 그 내부의 어느부분이 문제인지 조금 더 특정할 수 있을 것이다.

profiler를 통해 얻을 수 있는 이점

결과적으로 profiler를 통해 얻을 수 있는 이점은 다음과 같다.

  1. 해당 페이지가 rendering되는 전체 시간의 파악이 가능하다. (User가 기다려야 하는 시간 확인가능)
  2. 해당 페이지에 불필요한 re-render로 인한 시간 및 성능 문제가 발생하는지 파악할 수 있다. (만일 있다면 React.memo 등으로 처리)
  3. 해당 페이지 내부의 컴포넌트 중 어느 컴포넌트의 문제로 인함인지 파악하여, 내부의 컴포넌트로 들어가 더 세부적인 원인을 파악할 수 있다.

삼항연산을 통한 게시글 수정 로직 (2021-04-27 화)

게시글 수정과정

어떤 서비스의 코드를 작성하기전에 그 서비스가 어떻게 진행될지 생각해보고 로직을 짜는 과정은 꽤나 중요하다고 생각된다.

이에 이번에 진행했던 게시판 프로젝트 중 게시글 수정로직을 정리해보려 한다. (아래사진은 로직을 고민한 초안)

1) 유저확인

게시글의 작성자와 현재 수정을 진행하려하는 유저가 동일한지 확인

2) 상황에 따른 과정 분기

유저확인이 끝났다면, 유저가 어떤 수정을 진행할지에 대해 분기하여 생각해본다. 이때 게시글, 제목을 수정하는 것은 간단하기에 제쳐두고 생각.

  1. 제목, 내용, 작성자, 작성일 출력(default)
  2. 첨부파일 유무에 따른 분기
    1. if(file) -> 파일 출력
      1. 유저가 file 수정시 수정된 file로 axios.post
      2. 유저가 file 삭제시 file 비워서 axios.post
      3. 유저가 file은 수정하지 않았을 시 원 상태 그대로 axios.post
    2. if(!file) -> 파일이 없다는 문구 출력
      1. 유저가 file 추가시 추가한 file 넣어 axios.post
      2. 유저가 file 수정하지 않았을 시 빈 상태 그대로 axios.post

3) 분기한 상황 묶어보기

유저의 수정진행에 따라 상황을 나누었다면 해당 상황들 중에 묶어서 생각할 수 있는 것들을 살펴본다.

이때 server의 API 명세서에 따라 file을 수정했는지를 나타내는 file_status와 (수정시 1, 미수정시 0) file 자체를 나타내는 file로 나누어 정리한다.

  1. if(file)
    1. { file_status: 0, file: isIncluded }
    2. { file_status: 1, file: isIncluded }
  2. if(!file)
    1. { file_status: 0, file: undefined }
    2. { file_status: 1, file: undefined }

4) 묶음을 기준으로 실제 보낼 데이터 확인

3번의 과정을 통해 상황에 따른 API 명세를 묶어 생각했다면, 실제로 axios.post시에 보내야 할 data들을 생각해본다.

4-1) file을 변경한 경우

API 명세에 따르면 file-status: 0인 경우 file에 어떤 데이터를 넘겨도 반영되지 않으므로, file-status만 신경써서 넘기면 된다.

4-2) file을 삭제한 경우

유저가 file을 삭제한 경우 file-status는 1, file은 {}이 된다. 또한, 이미 file이 존재하지 않은 경우에도 file은 {}가 되고 file-status만 0의 차이를 보이는데,

위에서 설명한 바와 같이 (4-1의 과정) 변경하지 않은 경우에는 어차피 file이 반영되지 않으므로 신경쓰지 않아도 괜찮다.

4-3) file을 대체 혹은 새로 삽입한 경우

사실상 4-1, 4-2의 과정은 간단한 예외처리와 같은 부분이라면 실제적으로 수정이 들어가는 부분은 이번 과정이라고 볼 수 있다.

file을 다른 file로 수정한 경우와 file이 이전에 없었는데 새로 삽입하는 경우는 동일하게 볼 수 있기에 함께 묶어서 과정을 작성할 수 있다.

이 경우 역시 이미 있는 file에 대한 변경사항이 없다면 file에 무엇이 들어가도 상관이 없으므로 변경될 file만 반영하면 된다.

사실 이 과정에서 예외사항이 발생할 수 있기에 골치가 아픈데 이처럼 미리 분기될 부분들을 정의하면 더욱 간단히 묶어서 끝낼 수 있다.

5. view 부분에 삼항연산 작성

axios를 이용한 서버와의 통신부분 코드가 완료되었다면 보여지는 view component에 대한 작성을 할 차례.

react의 render부분이 JSX라는 특성상 if 또는 switch구문이 아닌 삼항연산자를 이용해 해당부분에 대한 분기가 필요한데, 삼항연산 로직을 작성할 때에는 순서가 중요하다.

먼저, 원래 file이 없었던 경우 (file을 수정하지 않았고, filepath가 존재하지 않으면) “첨부파일이 없습니다.”를 출력한다. 여기서 “filepath가 존재하지 않으면”을 &&로 묶어주지 않으면, 파일이 있어도 없다고 출력되므로 유의.

이후 file을 변경한 경우(file_status가 1) 새롭게 들어온 file을(previewURL) 미리보기로 보여주고, 변경 전에 원래 file이 존재한 경우 해당 file의(filepath) 미리보기를 보여준다.

게시글에 image를 첨부하여 서버로 보내고, 받아왔을 때, image의 주소가 base64형식으로 되어있다면, decode를 해야 사용할 수 있다.

물론 서버가 어떻게 되어있느냐에 따라 다르지만, 만일 서버에서 받아온 image의 주소를 그대로 넣어도 image가 뜨지 않을 경우 시도해볼만 하다.

만일 decode가 필요한 상황이라면 위 변수 decodeURI 자체를 웹주소에 넣어도 해당 image가 출력되므로, 넣어보는 방식으로도 확인해볼 수 있다.

파일 업로드를 위한 form과 blob (2021-04-23 금)

form data

클라이언트에서 서버로 데이터를 보낼때 text만을 이용한다면 딱히 form으로 보내거나 Content-Type을 설정하는데에 신경을 쓸 필요가 없었음. 그래서 여지껏은 form을 이용해볼 기회가 없었는데, 게시판에 첨부파일 기능이 필요하면서 이용해보게됨.

처음 form을 사용할 때 기억해야 할 부분

form으로 데이터를 보낼 때 계속해서 오류가 발생했었는데, 이유는 데이터가 들어있는 form 자체를 보내야 한다는 것을 자꾸 기억하지 못하기 때문.

이처럼 formData를 선언 후, 보낼 데이터들을 formData에 넣어주면, 당연히 formData만 보내주면 되는데, 이 것이 익숙치 않으면 헷깔릴 수 있음.

파일을 보낼 때 Blob 변환

다음으로 파일을 보낼 때 blob으로 변환하는 부분. 이 부분은 서버가 어떻게 구현되어 있느냐에 따라 다르겠지만, 일반적으로 대용량 파일을 위해서 blob을 사용하는 경우가 많다고 함.

이때, header의 Content-Type을 multipart/form-data로 지정해줘야 하며, 만일 파일이 없을 경우에는 빈 객체를 넣어 보내주어야 오류가 나지 않음. (물론 이 부분 역시 서버의 구현에 따라 다르므로 API를 참조하거나 백엔드 개발자와 상의가 필요함.)

blob 역시 formData처럼 new Blob()이라는 new 생성자를 통해 instance를 생성하여 사용함. 이때, 넣을 파일이 한개이더라도 []을 통해 배열을 넣어주어야 함. (form에서 데이터는 갯수에 상관없이 항상 배열화되어 넘어가야 함)

proxy를 통한 CORS 해결 (2021-04-21 수)

CORS 에러

4월 19일 월요일부터 게시판을 만드는 mini-project를 시작했는데, CORS에러와 함께 client side에서 cookie를 받아오지 못하는 등의 에러들로 고생을 했다.

서버는 RestAPI로 구축이 되어있는 상황에서 API 명세서를 확인하며 axios로 기능을 구현했으나, CORS에러가 나타나며 로그인 후 cookie를 client side에서 받아오지 못했다.

처음에는 코드의 로직 자체에 문제점이 있다고 판단했으나, 개발자도구를 살펴보니 network탭의 Headers에는 cookie가 잘 실려 오지만, application탭에서는 cookie가 존재하지 않았고, 오늘까지 장장 3일에 걸친 이를 해결하기위한 과정이 시작되었다.

기본적으로 CORS는 위와같이 Server와 Client의 주소가 다를때 나타나는데 이를 해결하기위한 구글의 방법을 찾아본 결과 아래의 방법이 대표적으로 나왔다.

withCredentials: true

이 방법은 cross-site Access-Control 요청에 인증서를 포함시킴으로서 CORS를 해결하는 방법이다. 따라서 client side에서는 위와 같이 true로 해줌으로서 조치할 수 있다.

다만, 이 방법을 사용하려면 server side에서도 credential을 true로 설정해줘야 한다.

proxy 설정

헌데, 위 server side를 고칠 수 없는 상황이라 위 방법을 쓸 수 없다면 어떻게 해야할까? 이럴때 사용할 수 있는 방법이 proxy를 이용하는 것이다.

proxy는 보안상의 이유로 직접 통신할 수 없는 두 점 사이에서 대리로 통신하는 중계역할을 하는 것을 말한다. 따라서 client 입장에서 proxy는 원격 server로 볼 수 있고, server 입장에서는 원격 client로 볼 수 있다.

proxy없이 요청을 하게 될 경우 server와 client의 uri가 다르기에 CORS에러가 발생되지만, proxy가 server와 client사이의 중계기능(가상서버) 역할을 할때

client는 proxy에게 client uri로 그대로 요청을 하고 proxy내부에서 이를 server uri로 변환하여 server로 보내는 방식으로 이루어지기 때문에

요청도 제대로 들어가고 Host와 Origin이 같으므로 CORS문제도 없게된다. (아래사진참조)

cookie 관리하는 방법 (2021-04-20 화)

session과 cookie

로그인한 유저의 인증방식 중 session방식은 로그인한 시간을 기준으로 만료날짜까지의 시간동안 인증이 되는 방식이다. 따라서 로그인을 요청하면 server에서 session을 cookie에 담아준다.

session을 cookie에 담아주면 브라우저에 저장이 되기 때문에, 로그아웃시에 client side에서 이를 delete하는 로직이 필요하다. (나는 client side의 정보를 파기하는 것은 생각치 못했는데, 이로인해 지속적으로 문제가 발생되었었다.)

이를 관리하기 편한 방법은 library를 사용하는 것인데, 구글링을 해보면 대부분 react-cookie 또는 js-cookie를 추천하는 편이다. 나의 경우 js-cookie를 이용했다.

기본적으로 server에 로그인을 요청하면 cookie를 함께 보내주지만, 혹시나 client side에서 cookie를 생성해야 할 경우 아래의 코드를 이용한다.

cookie를 삭제할 경우 해당 name을 메소드에 넣어주는 것만으로 삭제가 된다.

cookie에 대해

처음에 나는 cookie로 들어오는 값을 따로 변수에 담아 필요시 이용하려했는데, cookie는 server에 요청을 날릴때 필요하면 자동으로 날아간다.

따라서 관리보다는 필요에 따라 적절히 삭제하는데에 중점을 둬야 한다.

react에서 파일올리기 기능 구현 (2021-04-16 금)

file upload 기능

서비스 및 어플리케이션을 만들다보면 파일을 업로드하는 기능이 필요할 때가 있다. 헌데, 파일을 업로드하는 과정은 어떻게 이루어지는지를 전혀 몰라서 찾아보게 되었다.

결과적으로 보았을 때, 우리가 input의 내용을 local state에 담아 axios.post를 통해 서버로 보내는 방식은 동일한데, 그저 보내는 data의 타입이 text에서 file로 바뀐 것일 뿐으로 보인다.

input type=”file”

평소 회원가입, 로그인 등의 기능을 구현하는 과정에서 input의 type="text"로 했던 부분을 type="file로 하면 file을 담을 수 있는 input이 된다.

1) accept

input의 메소드로 accept를 사용하면 user가 입력할 수 있는 파일의 유형을 선택할 수 있다. 그러나 이 것은 강제는 아니고, 입력할때의 default option을 정해주는 것이다. 따라서 파일을 입력할때 user가 모든 파일 (*.*)로 변경할 수 있다.

2) multiple

input의 메소드로 multiple을 사용하면 파일 선택시 여러개의 파일을 선택할 수 있다. default는 false로 하나의 파일만 선택이 가능하다. 이 경우에는 state에 file 하나가 아닌 file들이 담겨있는 배열이 들어가므로 이를 감안한 코드 작성이 필요하다.

new FormData()

일반적으로 form 태그를 사용하게 될 경우 onClick시에 페이지 전환이 일어나므로 event.preventDefault()를 이용해 전환을 막아야 한다.

이런 번거로운 부분을 피하고, 페이지 전환이 없이 form data를 보내고 싶을때 사용하는 것이 FormData이다.

1) formData에 data 넣기

formData(instance)는 append 메소드를 사용해 key-value쌍을 추가할 수 있다. 이때 key는 중복이 되어도 value가 바뀌지 않으며, 동일한 key에 새로운 value로 추가되는 방식으로 동작한다.

2) formData의 data 확인하기

formData(instance)에 들어있는 data는 특정 key가 존재하는지 확인, key에 해당하는 첫번째 value 확인, key에 해당하는 모든 value를 확인. 의 세 가지 방법을 사용해 확인할 수 있다.

custom hook의 확장성

custom hooks를 공부하고 사용하는 과정에서 custom hook이란 결국 setState를 하는 로직을 custom하는 것이라고 생각했는데, 그 이상으로 생각해야 함.

custom hooks에는 꼭 [state, setState]만이 아니라 내가 필요로 하는 다른 것도 만들어 넣고 비구조화 할당을 통해 가져올 수 있음.

아래는 input에 대한 state관리와 reset을 할 수 있는 함수를 가진 custom hook 예시임.

이 외에도 추가적으로 더 필요한 부분들이 발생한다면 해당 기능에 대한 함수를 구현하여 return 후 사용하면 됨.

또한 기본적인 useState같은 hook들은 const [state, setState] = useState(initialState)와 같이 배열로 비구조화 할당을 했던 것과 달리 위 예시처럼 객체로 반환할 경우 객체로 비구조화 할당을 진행해야 하는데,

이때, 사용하는 변수명을 { 설정값: 사용할이름}의 방식으로 지정해줄 수 있음.

이외에 공부한 점

apollo-client에서 query의 variables

apollo-client에서 query 및 mutation을 날릴때 variables가 필요한 떄가 있음. 예를들어 list의 특정 요소를 Delete를 할 때에는 해당 요소의 id가 필요하기 때문에 variables로 id가 필요함.

이 경우 variables의 값으로 객체 {}가 들어감을 잊지 말아야 함.

또한 서버에서 id와 같은 값을 Int형으로 요청할경우 통신 후 json으로 인해 String으로 변환될 수 있다는 점을 인지하여 parseInt()를 씌워주는 것도 잊지 말자.
